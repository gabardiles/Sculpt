Methodology · Grounded Generation

Creative freedom,
zero hallucination.

How we let an AI compose an app — its programs, its screens, its content — while guaranteeing it can never invent something that doesn’t exist. The trick isn’t a smarter model. It’s grounding the model in real source-of-truth, and putting a validation gate between “suggested” and “shipped.”

The idea in one line

Truth is enforced. Creativity is invited.

Hallucination happens when a model is asked for a fact it doesn’t have, so it fabricates one. We remove that failure mode entirely by never trusting the model for facts — only for arrangement. Every concrete thing the model references (a database column, a UI component, an exercise, a design token) must already exist in a document we control. Anything it names that isn’t in those documents is dropped before it ever touches the app.

That leaves the model free to do the thing it’s genuinely great at: choosing, sequencing, naming, and combining real building blocks into something tailored and new.

The pipeline

Six steps from intent to committed artifact

1
INPUT

Capture the brief

A small, structured set of choices — goal, route, days, equipment, level. Bounded options, not free prose, so intent is unambiguous from the start.

2
GROUND

Load source-of-truth

Before generating, we hand the model the real vocabulary: the schema, the component/exercise registry, the design tokens, the domain rules. It can only build from what it’s shown.

3
GENERATE

Compose against a strict contract

The model returns only a fixed JSON shape — no prose, no surprises. Inside that shape it has real freedom: which blocks, in what order, how grouped, what to call it.

4
NORMALIZE

Clamp & bound

Numbers are clamped to safe ranges, strings length-capped, unknown fields stripped. The output becomes well-formed regardless of what came back.

✓
VALIDATE · the gate

Reconcile against reality

Every referenced entity is looked up in the source-of-truth by exact key. No match → it’s dropped. A hallucinated exercise name simply isn’t found, so it never survives. This is the step that makes the guarantee real.

5
COMMIT

Persist only what passed

Only validated, ID-resolved artifacts get written. The database never holds a thing the model invented — it holds real rows pointing at real records.

● suggested freely (steps 1–3) ◆ made safe (4) ✓ proven real (the gate) ▣ shipped (5)
The enforced documents

What we make the model answer to

Each document is a contract. Some constrain what can exist (data & components), some constrain how it must look (shape & design), some constrain what makes sense (domain rules). Together they leave a precise, creative gap for the model to fill — and nothing more.

DocumentWhat it isPurpose · what it prevents
Schema / migrations The data contract — tables, columns, types. Output maps to real columns only. Prevents invented fields.
Component / exercise registry The catalog of valid building blocks, each with a stable ID. The model picks from real names; we resolve to IDs. Prevents invented components.
Design system / tokens Colors, fonts, spacing, primitives (GlassCard, PillButton…). Visuals stay on-brand and buildable. Prevents invented styles.
Output contract (JSON schema) The exact response shape the model must return. Parseable, predictable structure. Prevents freeform drift.
Domain rules / guardrails The logic of the field — phases, progression, same-muscle swaps. Combinations make sense, not just parse. Prevents plausible-but-wrong output.
Conventions (CLAUDE.md) How this codebase is written — patterns, naming, idioms. New code reads like the old code. Prevents stylistic drift.
The guarantee

Why it’s ~0% hallucination

The model is allowed to suggest anything. But only things that already exist can be committed. The validation gate sits between the two — so a hallucination is harmless: it’s caught at lookup and discarded.

① The model may suggest…

  • which blocks to use & in what order
  • how to group and name things
  • volume, emphasis, narrative tone
  • anything — even a made-up name
VALIDATION GATE

② Only the real gets committed

  • names resolved to real IDs by exact match
  • unmatched references → silently dropped
  • values clamped to safe ranges
  • writes reference real rows only
Concretely: if the model returns an exercise called “Hyperbolic Glute Cannon,” the server looks it up in the exercise library, finds nothing, and drops it. The user’s program is built only from movements that truly exist. The fabrication never reaches the database — or the screen.
The other half

Where the freedom actually lives

🧩 Composition, not invention

Think of a chef with a fixed, real pantry. The ingredients are fixed; the dishes are infinite. The model’s creativity is in the recipe — selection, balance, sequence — never in conjuring an ingredient that isn’t on the shelf.

🎚️ Tailoring within bounds

Two athletes, two briefs, two genuinely different programs — same validated library. The freedom is real and visible to the user, while the substrate stays provably true.

✍️ Naming & voice

Titles, summaries, focus lines, cues — the human-feeling layer — are free text the model owns. They’re cosmetic, so they carry no factual risk and need no lookup.

🔁 Cheap to rebuild

Because the contract is strict and the gate is automatic, regenerating is one call. The user can “rebuild” until a composition clicks — every result is already guaranteed valid.
